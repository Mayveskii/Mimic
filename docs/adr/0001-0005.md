# ADR 0001: Mimic is an MCP Server

## Context

Mimic needs to be accessible to AI-agents. Two options: embed into an existing MCP client as a plugin/fork, or implement as a standalone MCP server.

## Decision

Mimic = MCP server. Any MCP client connects externally.

## Rationale

- MCP protocol already exists — no need to fork anything
- CGO from Go to C is trivial, N-API from TypeScript to C is fragile
- Single binary = single deploy, no dependency on client version
- Any MCP client can connect, not just one specific client

## Consequences

- Need to implement MCP transport (stdio/SSE/HTTP)
- Dependency on MCP protocol spec (stable)
- Mimic cannot directly call client internal APIs (and should not)

# ADR 0002: C-core Inside MCP Server via CGO

## Context

C-core can run as a separate process (gRPC/Unix socket) or link into the MCP server via CGO.

## Decision

C-core links into the MCP server via CGO.

## Rationale

- Separate service = RTT per exec (~1-10ms per hop)
- CGO call < 1μs (direct function call)
- Single binary = simpler deploy, simpler CI, fewer failure points
- exec_chain can contain 10+ operations — separate service = 10+ RTT

## Consequences

- CGO overhead for Go↔C struct conversion (μs, not ms)
- Build requires GCC + Go toolchain
- Cannot restart C-core without restarting the server

# ADR 0003: Two Knowledge Sources — Distillation and Mimicry

## Context

Mimic draws knowledge from two distinct source types:
1. Production repositories (repos-manifest.yaml, 90+) → distillation of best commits via git blame → survival index
2. Own repositories (Mayveskii/*) → behavior selection: how to implement functions that Mimic should have

## Decision

Separate sources into two manifests:
- `mimicrya/repos-manifest.yaml` — production repos for distillation (survival index → patterns)
- `mimicrya/behavior-sources.yaml` — own repos for behavior selection (mimicry: which functions to implement and how)

## Rationale

- Distillation = extracting proven patterns from production code (etcd, k8s, go-ethereum)
- Mimicry = selecting behavior from best implementations (exa-mcp-server = how to search, gh-aw-mcpg = how to route, code-mode = how to manage tool loop, bun = how to orchestrate phases)
- Different processes with different outputs: distillation → mesh slot, mimicry → function implementation in Mimic

## Consequences

- Need to track both manifests
- Behavior sources synced manually (user decides what to take)
- Distillation can be automated (cron)

# ADR 0004: Phase Graph Orchestration

## Context

How to orchestrate execution: blind (received command → executed) or with phases (classify → plan → validate → exec → verify).

## Decision

Phase graph: CLASSIFY → PLAN → VALIDATE → EXEC → VERIFY → RESPOND, with rollback capability.

## Rationale

- Extracted from Mayveskii/bun (PR #30412): 26 workflow files with phase transitions
- Blind execution = risk of conflict, budget overrun, data corruption
- Rollbacks (VALIDATE fail → PLAN) allow chain rebuild without error
- VERIFY phase (2-vote) prevents propagating mistakes

## Consequences

- Every request passes 6 phases
- VALIDATE = O(n²) for chain of n operations (acceptable for n < 100)
- Need to determine which operations require VERIFY

# ADR 0005: libbmap.a Without Sources — Rewrite

## Context

libbmap.a contains 39 functions but .c sources are missing — only .o files.

## Decision

Rewrite all 39 functions from scratch, based on:
1. Signatures from nm (argument types from headers if available)
2. Behavior from tests (if any)
3. Semantics from SEMANTICS.md

## Rationale

- Cannot modify what cannot be rebuilt
- Cannot guarantee correctness without sources
- Cannot debug problems in .o files

## Consequences

- Significant work (39 functions)
- Functions must be backward-compatible by signature
- z_density_compute — critical function, exact formula unknown, needs experimental determination
